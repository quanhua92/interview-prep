# Production Security Fundamentals -- Working Checklist

Use this checklist each time you practice. Check items as you verbalize them.
The goal is to work through ALL items without referring to discussion.md.

## Phase 1: Requirements (2-3 min)
- [ ] State the problem back to the interviewer in your own words
- [ ] Ask clarifying questions:
  - [ ] What is the deployment environment? (EC2, EKS, GKE, multi-cloud?)
  - [ ] Are there compliance requirements? (SOC 2, PCI-DSS, HIPAA?)
  - [ ] What is the trust boundary? (service-to-service, external-facing, third-party integrations?)
- [ ] List security requirements (3-5 items):
  - [ ] All user input validated and sanitized at the application layer
  - [ ] Secrets stored in a dedicated secrets manager with automatic rotation
  - [ ] Service-to-service communication authenticated via mTLS
  - [ ] IAM roles scoped to minimum required permissions per service
  - [ ] Container images verified against signed provenance before deployment
- [ ] List non-security non-functional requirements (latency, availability, scale):
  - [ ] Secrets fetch adds <50ms to startup (cached after first call)
  - [ ] mTLS handshake overhead absorbed by sidecar proxy (no app code change)
  - [ ] SBOM generation in CI adds <2min to build pipeline
  - [ ] IAM policy changes propagate within minutes
- [ ] State your threat model (attack vectors, blast radius):
  - [ ] External attacker: injection, SSRF, credential theft
  - [ ] Compromised dependency: supply chain attack (Log4Shell pattern)
  - [ ] Insider / compromised pod: overpermissioned IAM role expands blast radius
  - [ ] Blast radius bounded by per-service IAM roles and network segmentation

## Phase 2: High-Level Design (3-5 min)
- [ ] Draw a simple box diagram: Client -> LB -> [API Service] -> [Secrets Manager / Vault] -> [DB + Cache]
- [ ] Identify the main security enforcement points:
  - [ ] Load balancer: TLS termination, security headers (HSTS, CSP, X-Frame-Options)
  - [ ] API gateway / middleware: authentication (JWT validation), rate limiting on auth endpoints
  - [ ] Application layer: authorization (object-level checks), input validation, parameterized queries
  - [ ] Service mesh (Istio/Linkerd): mTLS between services, automatic cert rotation
  - [ ] Infrastructure: IAM roles per service, security groups, VPC segmentation
- [ ] Note the trust boundaries and where each defense applies:
  - [ ] External -> LB: TLS 1.2+, security headers, rate limiting
  - [ ] LB -> Service: internal network, mTLS if service mesh
  - [ ] Service -> Secrets Store: IAM-based access, no long-lived credentials
  - [ ] Service -> DB: credential from secrets manager, parameterized queries
  - [ ] CI -> Registry -> Cluster: signed images, admission webhook verification
- [ ] Present the diagram and get alignment

## Phase 3: Core Component Design (5-8 min)
- [ ] Explain injection prevention and justify parameterized queries:
  - [ ] Input sanitization fails: context-dependent escaping, ORM bypass patterns, partial attacker control
  - [ ] Parameterized queries fix at architectural level: query structure compiled first, data bound as typed values
  - [ ] Command injection: use `subprocess.run()` with `shell=False`, pass args as list
  - [ ] ORM trap: `text(f"...{user_input}")` bypasses ORM protection; use bind parameters with `text()`
- [ ] Define authorization enforcement points:
  - [ ] Authentication (who are you?) vs Authorization (what can you do?) are separate enforcement points
  - [ ] IDOR prevention: always fetch with `WHERE id = ? AND owner_id = ?` (or equivalent RBAC check)
  - [ ] Use UUIDs for external-facing IDs to prevent enumeration attacks
  - [ ] Audit both read and write paths separately for authorization gaps
- [ ] Sketch the secrets management architecture:
  - [ ] Source of truth: AWS Secrets Manager (AWS-native) or Vault (cross-cloud, dynamic secrets)
  - [ ] For K8s: External Secrets Operator syncs from store to K8s Secret on configurable interval
  - [ ] Pod reads K8s Secret as mounted file; rotation propagates without redeploy
  - [ ] Scan git history with gitleaks/trufflehog in pre-commit hooks and CI
- [ ] Walk through the IAM design:
  - [ ] One IAM role per K8s ServiceAccount per namespace (IRSA) or per GKE service account (Workload Identity)
  - [ ] No long-lived access keys; short-lived tokens via OIDC token exchange (1-hour default)
  - [ ] Scope: minimum actions on minimum resources (e.g., `dynamodb:GetItem` on specific table ARN, not `dynamodb:*` on `*`)
  - [ ] Use IAM Access Analyzer to identify over-provisioned policies

## Phase 4: Deep Dive (5-10 min, interviewer-directed)
- [ ] Discuss supply chain security:
  - [ ] SBOM (CycloneDX or SPDX) generated by Syft: machine-readable inventory of all direct + transitive dependencies
  - [ ] Dependabot/Renovate: auto-merge patch updates after CI passes, group minor updates weekly, surface majors for human review
  - [ ] Sigstore/cosign: sign images with OIDC-backed ephemeral key tied to CI pipeline identity
  - [ ] Admission webhook (Kyverno/OPA Gatekeeper): reject unsigned images
  - [ ] SLSA levels: L1 (documented build) -> L2 (signed provenance) -> L3 (hardened build infra) -> L4 (hermetic, reproducible)
- [ ] Explain SSRF mitigation:
  - [ ] IMDSv2: requires PUT for session token before GET; breaks basic SSRF via redirect
  - [ ] URL allowlisting: explicit permitted hostnames/IP ranges; block RFC 1918 and 169.254.x.x
  - [ ] Egress filtering: security groups, VPC ACLs, or egress proxy block metadata endpoint
  - [ ] Capital One breach pattern: SSRF -> IMDS -> overpermissioned IAM role -> data exfiltration
- [ ] Explain mTLS for service-to-service:
  - [ ] TLS authenticates server only; mTLS authenticates both sides
  - [ ] Service mesh (Istio/Linkerd) handles mTLS transparently via sidecar proxies
  - [ ] Istio: 24-hour cert lifetime, SPIFFE-format identity URIs, automatic rotation
  - [ ] Application code sees plaintext; no TLS code changes needed
- [ ] Address common production failure modes:
  - [ ] Secrets in logs: log-scrubbing middleware redacts sensitive field names before log sink
  - [ ] Verbose errors in API responses: return generic error IDs, log details internally
  - [ ] Missing security headers: set HSTS, X-Content-Type-Options, CSP at load balancer level
  - [ ] Default credentials on infrastructure: change defaults before instance is network-reachable
  - [ ] Over-broad CORS: explicit origin allowlist; never `*` on authenticated endpoints

## Phase 5: Wrap-Up (2-3 min)
- [ ] Summarize the security design in 2-3 sentences
- [ ] State the trade-offs you made and why (e.g., ESO vs Vault Agent, SLSA L2 vs L3 investment)
- [ ] Mention what you would improve with more time (e.g., full SLSA L3 pipeline, automated policy generation)
- [ ] Ask the interviewer if they have questions
