# Resume Structure

## Overview
How to build a one-page resume that passes both ATS systems and recruiter screening, using structured sections, quantified bullet points, and strategic ordering of information.

## Best Practices
- Keep your resume to one page unless you have 10+ years of highly relevant experience
- Use a standard, clean font (Calibri, Garamond, Helvetica) at 10-11pt for body text
- Start every bullet point with a strong action verb (Led, Built, Reduced, Designed, Shipped)
- Quantify results wherever possible -- use numbers, percentages, dollar amounts, and timeframes
- Tailor your resume for each role by reordering bullets and adjusting keywords to match the job description
- Use reverse chronological order for experience and education sections
- Save as PDF to preserve formatting -- never send a .docx unless specifically requested
- Include relevant keywords from the job description to pass ATS screening

## Structure / Section Order
| Section | What to Include | Placement Notes |
|---------|----------------|-----------------|
| Header | Name, phone, email, LinkedIn, GitHub/portfolio URL | Always at the top; no photo or home address |
| Summary | 2-3 sentence overview of who you are and what you offer | Optional but recommended for career changers or senior roles |
| Experience | Company, title, dates, 3-5 bullets per role | Most important section -- put it first unless you are a recent grad |
| Education | Degree, school, graduation year, relevant coursework or honors | Recent grads can place this above experience |
| Skills | Grouped by category (languages, frameworks, tools, domains) | Keep it honest -- you will be asked about anything listed |

## Bullet Writing Framework (X to Y to Z)
Every strong bullet follows this pattern: **Action + What You Did + Result**
- "Led (action) the migration of a monolith to microservices (what) which reduced API latency by 40% and improved deploy frequency from weekly to daily (result)"
- Avoid: "Responsible for managing server infrastructure" -- this is a job description, not an achievement

## Good vs Bad Examples

**Bad bullet:** "Worked on the frontend team building React components."

This is bad because it says nothing about impact, scope, or outcome. It reads like a job posting, not evidence of capability.

**Good bullet:** "Built a reusable component library of 30+ React components, reducing feature development time by 35% across four product teams."

This is good because it specifies what was built (30+ components), the scope (four teams), and the measurable result (35% faster development).

**Bad summary:** "Passionate software engineer with 5 years of experience looking for new opportunities."

This is bad because it is generic and could apply to thousands of engineers. It wastes valuable space with filler language.

**Good summary:** "Full-stack engineer with 5 years of experience building high-traffic web applications at scale. Specialized in React, Node.js, and cloud infrastructure on AWS. Led two team migrations to microservices and reduced deployment time from 2 hours to 15 minutes."

## Common Mistakes
- Listing responsibilities instead of achievements -- "Responsible for X" tells the reader nothing about how well you did it
- Using generic summaries with empty adjectives like "passionate," "hardworking," or "self-motivated"
- Including outdated or irrelevant skills just to fill space (e.g., listing HTML if you are a senior backend engineer)
- Using dense paragraphs instead of scannable bullet points -- recruiters spend 6-7 seconds on initial review
- Over-designing with colors, graphics, or unusual layouts that confuse ATS parsers
- Leaving gaps unexplained or using vague date ranges (e.g., "2020-Present" when you mean "2020-2023")
